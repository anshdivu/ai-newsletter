Tips & Tricks
Keep CLAUDE.md under 200 lines · Claude Code best practices
Shorter wins. Frontier models reliably follow about 150 to 200 instructions, and Claude Code's system prompt already spends roughly 50. Past that, your rules start getting dropped. Cut CLAUDE.md to universal commands, architecture, and conventions. Point to deep docs with file references instead of pasting them. Best practices →
If it must happen every time, make it a hook · SmartScope
CLAUDE.md is advice. Hooks are guarantees. Claude follows the markdown about 80 percent of the time. A hook fires 100 percent. Move anything non-negotiable, a format-on-save, a required test gate, a commit check, out of CLAUDE.md and into a hook. Keep the markdown for guidance, not promises. 11 techniques →
Install an LSP plugin first · Composio
The single highest-impact plugin you can add. An LSP plugin hands Claude automatic diagnostics after every edit. Type errors, unused imports, missing returns, caught the moment they appear instead of three steps later. It turns silent mistakes into instant feedback. Plugin picks →
Engineer the context, not the wording · The AI Corner
Context engineering replaced prompt engineering as the real lever. What you load, and in what order, matters more than how you phrase the ask. Build the context first. An identity file, a voice profile, your stack. Load them once through global instructions. Then prompt. Power-user guide →

CLAUDE.md, the file Claude reads before your code
CLAUDE.md is the memory file Claude Code loads at the start of every session. Commands, architecture, conventions, the things you would tell a new hire on day one. It is advisory, not binding. Claude honors it about 80 percent of the time, so length is the enemy. Models track maybe 150 to 200 instructions well, and the system prompt already eats 50. Keep it tight. List the universal rules, link out to deep docs with file references, and never repeat what a linter already enforces. For the rules that must never slip, reach for hooks instead.
